# План: dropdown-фільтр жанрів для ігор і кіно

## Як я зрозуміла задачу

Потрібно змінити UX поля `Жанри` у пошуку/фільтрах для двох екранів:

- `Ігри`
- `Кіно`

Зараз поле `Жанри` працює як звичайне текстове поле вводу. Його треба замінити на dropdown-multiselect.

## Цільовий результат

Поле `Жанри` має працювати так само, як зараз працює фільтр `Платформа`:

- це випадаючий список;
- у списку є чекбокси;
- можна обрати кілька значень;
- є опція `обрати всі`;
- збережений поточний UX відкриття/закриття списку;
- візуально і поведінково це той самий патерн, що й `Платформа`.

## Ключова вимога

Потрібно не просто локально скопіювати логіку `Платформи`, а перевикористати її.

Якщо фільтр `Платформа` ще не винесений в окрему shared-компоненту, треба:

1. винести його в окрему компоненту;
2. зберегти поточну поведінку;
3. використати цю саму компоненту і для `Платформи`, і для `Жанрів`.

## Що саме змінюємо

### 1. Games

На екрані `Ігри`:

- прибрати текстове поле `Жанри`;
- замінити його на dropdown з чекбоксами;
- джерелом опцій зробити список наявних жанрів;
- підключити той самий shared multiselect, який використовується для `Платформи`.

### 2. Films

На екрані `Кіно`:

- прибрати текстове поле `Жанри`;
- замінити його на dropdown з чекбоксами;
- джерелом опцій зробити список наявних жанрів;
- використати той самий shared multiselect-патерн.

## Джерело даних для жанрів

Список жанрів не має бути хардкодом у UI.

Очікування:

- опції будуються з наявних жанрів у відповідному домені;
- `games` і `films` можуть мати різні набори жанрів;
- список має бути достатньо стабільним для фільтра;
- UX не повинен змушувати користувача вручну вгадувати текст жанру.

Під час реалізації потрібно окремо визначити, що є правильним джерелом:

- нормалізовані genre relations;
- агреговані жанри з item metadata;
- інший уже наявний шар, якщо він у проекті є правильнішим.

## Поведінка компоненти

Shared multiselect-компонента має підтримувати:

- `label`;
- масив `options`;
- вибрані значення;
- режим `all selected`;
- чекбокси для окремих значень;
- callback зміни;
- керування відкриттям/закриттям;
- стан disabled, якщо буде потрібний;
- можливість повторно використати її для інших catalog-фільтрів.

## Що важливо не зламати

- поточний UX фільтра `Платформа`;
- застосування фільтрів у `games`;
- застосування фільтрів у `films`;
- сумісність з уже наявним screen-context persistence;
- зовнішній вигляд і роботу інших dropdown-фільтрів, якщо вони поруч.

## Рекомендований напрям реалізації

### Етап 1. Виділити shared компоненту

Винести поточну логіку dropdown з `Платформи` у перевикористовувану компоненту.

### Етап 2. Перевести `Платформу` на shared компоненту

Спочатку підключити до неї вже існуючий фільтр `Платформа`, не змінюючи його поведінку.

### Етап 3. Додати `Жанри`

Після цього замінити текстове поле `Жанри` на той самий dropdown-multiselect:

- у `games`;
- у `films`.

### Етап 4. Перевірити UX

Переконатися, що:

- можна обирати кілька жанрів;
- працює `обрати всі`;
- фільтри не злітають;
- у двох екранах немає розходження в поведінці компоненти.

## Критерій готовності

Задача вважається виконаною, коли:

1. `Жанри` в `Іграх` більше не є текстовим полем.
2. `Жанри` в `Кіно` більше не є текстовим полем.
3. `Платформа` і `Жанри` працюють через один shared dropdown-multiselect патерн.
4. Користувач може вибирати жанри через список з галочками, а не вводити їх вручну.
